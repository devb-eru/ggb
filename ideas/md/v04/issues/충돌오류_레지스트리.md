## 2. ID 체계

### 2.1 형식

```text
GGB-CNF-YYYY-NNNN
GGB-ERR-YYYY-NNNN
```

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| `GGB` | 프로젝트 식별자 | GGB |
| `CNF` | 둘 이상의 문서가 서로 다른 규칙을 정의한 충돌 | [`GGB-CNF-2026-0001`](items/GGB-CNF-2026-0001_J4와_E3_4M_선행_순서_충돌.md) |
| `ERR` | 누락, 잘못된 ID, 저장 불능 등 단독으로도 잘못된 오류 | [`GGB-ERR-2026-0001`](items/GGB-ERR-2026-0001_마라_2_병합_분리_결과_저장_효과_누락.md) |
| `YYYY` | 최초 등록 연도 | 2026 |
| `NNNN` | 유형별 일련번호 | 0001 |

세부 문제가 독립적으로 수정·검증될 수 있으면 별도 ID를 발급한다. 하나의 원인에서 파생된 단순 증상은 새 ID 대신 해당 항목의 영향으로 기록한다.

### 2.2 상태

| 상태 | 의미 | 다음 상태 |
| --- | --- | --- |
| `OPEN` | 문제 확인, 수정 전 | IN_PROGRESS, DEFERRED, WONT_FIX |
| `IN_PROGRESS` | 영향 문서를 수정하는 중 | RESOLVED, BLOCKED |
| `BLOCKED` | 결정 또는 선행 수정이 필요 | IN_PROGRESS, DEFERRED |
| `RESOLVED` | 문서 수정 완료, 검증 전 | VERIFIED, IN_PROGRESS |
| `VERIFIED` | 회귀 검증까지 통과 | 재발 시 OPEN |
| `DEFERRED` | 의도적으로 후속 버전으로 연기 | OPEN |
| `WONT_FIX` | 문제를 인지하고 현 상태 유지 결정 | 재검토 시 OPEN |

`RESOLVED`는 수정 완료를 뜻하고, `VERIFIED`는 관련 흐름과 상태를 다시 검사해 해결이 확인됐음을 뜻한다.

### 2.3 해결 증거

항목을 `RESOLVED` 이상으로 변경할 때 다음을 기록한다.

```yaml
resolved_in:
  - 수정한 문서와 절
resolution_summary: 변경 내용
verification_ids:
  - QA 또는 정적 검사 ID
verified_on: YYYY-MM-DD
```

## 7. 상태 갱신 절차

이슈를 수정할 때 다음 순서를 사용한다.

1. 항목 상태를 `IN_PROGRESS`로 변경한다.
2. 기준 문서를 먼저 수정한다.
3. 영향 문서를 수정한다.
4. 해결 체크리스트를 갱신한다.
5. 수정이 끝나면 `RESOLVED`로 변경한다.
6. 지정된 QA ID를 실행한다.
7. 통과하면 `VERIFIED`로 변경하고 해결 이력에 기록한다.
8. 통합 문서 `00`, `README`, `18`은 마지막에 동기화한다.

재발 시 기존 ID를 다시 `OPEN`으로 변경한다. 같은 원인이 아니라면 새 ID를 발급한다.

## 8. 추가 권장 기능

### 8.1 문서별 소유자

각 이슈에 `resolution_owner`를 추가해 어느 문서를 먼저 수정해야 하는지 명시한다.

```yaml
resolution_owner:
  primary: "02"
  affected: ["03", "04", "17", "18"]
```

### 8.2 결정 기록

둘 이상의 해결안 중 하나를 선택할 때 `decision_id`를 발급한다.

```text
GGB-DEC-2026-0001
```

결정 기록에는 선택안, 기각안, 선택 이유와 영향 문서를 남긴다. 이후 같은 논의를 반복하는 일을 줄일 수 있다.

### 8.3 회귀 검사 매트릭스

각 이슈의 `verification_id`를 별도 QA 문서와 연결한다.

```text
이슈 ID
→ 수정 문서
→ 검증 경로
→ 기대 상태
→ 실제 결과
```

특히 최소 진행, 전원 완료, 저장 중 종료, 색 제거 접근성 경로를 고정 회귀 경로로 유지한다.

### 8.4 문서 기준선

검토 시점마다 다음을 기록한다.

- 검토 날짜.
- 기준 문서 버전.
- 가능하면 Git commit ID.
- 검사한 파일 목록.

문서가 변경된 뒤 예전 검토 결과를 최신 상태로 오해하는 일을 막는다.

### 8.5 자동 정적 검사

향후 간단한 검사 스크립트를 두는 것을 권장한다.

- 중복·미등록 이벤트 ID.
- 지도에 없는 `location_id`.
- 선언되지 않은 상태 플래그.
- `OPEN` 이슈 수 집계.
- 깨진 Markdown 링크.
- 코드 펜스와 Mermaid 블록 짝.
- `RESOLVED`인데 해결 증거가 없는 항목.

### 8.6 폐기·대체 관계

이슈가 다른 이슈에 흡수되면 삭제하지 않고 다음처럼 기록한다.

```yaml
status: WONT_FIX
superseded_by: GGB-CNF-YYYY-NNNN
```

이를 통해 ID와 변경 이력이 끊기지 않게 한다.
