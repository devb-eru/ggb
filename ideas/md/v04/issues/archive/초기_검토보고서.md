# GGB v0.4 충돌·오류 검토 보고서

> 보관본: ID 기반 운영 레지스트리 도입 전 작성된 최초 검토 결과다.

## 1. 문서 목적

본 문서는 `ideas/md/v04` 문서 세트를 교차 검토하여 발견한 진행 규칙, 상태 변수, 공간 ID, 결산 단계와 통합 문서의 충돌을 기록한다.

- 검토 범위: `README.md`, `00~18` Markdown 문서.
- 검토 기준: 최신 사용자 확정사항과 각 영역의 기준 문서.
- 현재 상태: 문제 식별과 권장 대처안만 기록했으며 원본 문서는 수정하지 않았다.
- 심각도:
  - `높음`: 진행 불능, 저장 손상, 핵심 분기 오작동 가능성.
  - `중간`: 구현 결과가 문서마다 달라지거나 후속 장면이 잘못 변형될 가능성.
  - `낮음`: 직접적인 진행 오류는 아니지만 제작·QA 과정에서 혼동을 일으킬 가능성.

## 3. 상세 발견 사항

### 3.1 J4와 E3_4M 순환 선행 조건

**심각도: 높음**

#### 문제

[03 전체 이벤트 흐름도](../../03_전체이벤트흐름도.md)와 [04 전체 이벤트 리스트·상태표](../../04_전체이벤트리스트_상태표.md)는 다음 순서를 사용한다.

```text
J4 완료
→ E3_4 완료 여부 확인
→ 미완료라면 E3_4M
→ 관계 결산
→ E5
```

그러나 [13 파열·전환·결산](../../13_이벤트상세_08_파열_전환_결산.md)의 J4 선행 조건은 `E3_4 또는 E3_4M`이다. 이 조건을 그대로 구현하면 다음 순환이 생긴다.

```text
J4를 시작하려면 E3_4M 필요
→ E3_4M을 시작하려면 J4 완료 필요
→ 최소 진행 경로 정지
```

[12 사용인 핵심 관계](../../12_이벤트상세_07_사용인핵심관계.md)도 E3_4M을 `E_HUB 조사 종료` 직후로 설명하여 J4 전후 순서가 명확히 고정되지 않았다.

#### 영향

- 에드가 핵심 관계 이벤트를 건너뛴 최소 진행 경로가 막힐 수 있다.
- 관계 이벤트 0명 경로의 QA 결과가 문서별로 달라진다.
- E5와 E6의 조건 검사 시점이 불명확해진다.

#### 권장 대처안

`03`과 `04`의 진행 순서를 정본으로 채택한다.

```text
E_HUB 조사 종료
→ J4
→ E3_4_complete 검사
→ false이면 E3_4M
→ E5
```

수정 기준:

- J4 선행 조건에서 `E3_4 또는 E3_4M`을 제거한다.
- E3_4M 선행 조건을 `J4 완료 && E3_4 미완료`로 고정한다.
- E5는 `J4 완료 && (E3_4 완료 || E3_4M 완료)`를 요구한다.

### 3.2 D5 이후 휴식과 S3 재생성 충돌

**심각도: 높음**

#### 문제

[02 루프·관계·기억·색상 서명 시스템](../../02_루프_관계_기억_색상서명시스템.md), [03 전체 이벤트 흐름도](../../03_전체이벤트흐름도.md), [04 전체 이벤트 리스트·상태표](../../04_전체이벤트리스트_상태표.md)는 다음 규칙을 사용한다.

- `BROKEN_RESET`은 D5 이후 한 번만 발생한다.
- 최초 `BROKEN_RESET`에서 S3 손상 월드를 생성한다.
- E1 이후의 수면은 `POST_BROKEN_REST`다.
- `POST_BROKEN_REST`는 수리 상태와 물리 배치를 유지한다.

반면 [17 상태 변수·이벤트 ID·Godot 데이터 구조](../../17_상태변수_이벤트ID_Godot데이터구조.md)는 `broken_reset_triggered=true`일 때 S3 기준 `loop_state`를 다시 생성하는 구조로 적혀 있다.

#### 영향

- E3_1~E3_5에서 수리한 장치가 휴식 후 되돌아갈 수 있다.
- E구간의 공간 상태와 관계 이벤트 재개 지점이 손실될 수 있다.
- `BROKEN_RESET`과 `POST_BROKEN_REST`가 같은 기능으로 잘못 구현될 수 있다.

#### 권장 대처안

후반 수면 상태를 명시적으로 분리한다.

```text
NORMAL_LOOP
→ FRACTURE_SLEEP
→ BROKEN_RESET_ONCE
→ POST_BROKEN
→ POST_BROKEN_REST 반복
```

- S3 생성은 `BROKEN_RESET_ONCE`에서만 허용한다.
- `POST_BROKEN_REST`는 시간·피로·대화 구간만 갱신한다.
- E구간 장치 수리, 문 상태, 관계 이벤트 진행도는 그대로 유지한다.

### 3.3 정상 리셋 트랜잭션 순서 불일치

**심각도: 높음**

#### 문제

[02 루프·관계·기억·색상 서명 시스템](../../02_루프_관계_기억_색상서명시스템.md)은 사용인 기억 압축 뒤 영구 상태를 커밋한다.

```text
영구 획득분 임시 저장
→ 사용인 기억 압축
→ 영구 상태 커밋
→ 물리 상태 폐기
```

[07 루프·영구 정보·숏컷](../../07_이벤트상세_02_루프_영구정보_숏컷.md)은 다음 순서를 공식 체인으로 정의한다.

```text
NORMAL_SLEEP
→ SYS_COMMIT
→ SYS_MEMORY
→ NORMAL_RESET
→ MORNING
→ ROUTE
```

#### 영향

- 기억 저장 실패 시 플레이어 영구 정보가 커밋됐는지 판단하기 어렵다.
- 비정상 종료 후 어느 단계부터 재개할지 문서마다 달라진다.
- 같은 관계 변화나 기록이 중복 적용될 수 있다.

#### 권장 대처안

`07`의 체인을 정본으로 채택한다.

- `SYS_COMMIT`: 플레이어의 영구 지식, 관계 결과, 일지와 퍼즐 검증 상태 저장.
- `SYS_MEMORY`: 사용인별 잔류 기억 저장.
- 두 단계가 모두 성공한 뒤에만 `NORMAL_RESET` 실행.
- 같은 `reset_transaction_id`로 재시도할 때 결과를 중복 적용하지 않는다.
- `02`와 `17`의 처리 순서와 복구 규칙을 이 체인에 맞춘다.

### 3.4 관계 결산 단계 불일치

**심각도: 높음**

#### 문제

[00 문서 목록·확정사항](../../00_v04_문서목록_및_확정사항.md), [02 루프·관계·기억·색상 서명 시스템](../../02_루프_관계_기억_색상서명시스템.md), [13 파열·전환·결산](../../13_이벤트상세_08_파열_전환_결산.md), [17 상태 변수·이벤트 ID·Godot 데이터 구조](../../17_상태변수_이벤트ID_Godot데이터구조.md), [18 통합본](../../18_GGB_게임기획서_v0.4_통합본.md)은 대체로 다음 판정을 사용한다.

```text
0~1명: LOW
2~3명: MID
4~5명: HIGH
5명: all_servants_complete 추가
```

[03 전체 이벤트 흐름도](../../03_전체이벤트흐름도.md)와 [04 전체 이벤트 리스트·상태표](../../04_전체이벤트리스트_상태표.md)는 다음 4단계 enum을 사용한다.

```text
0~1명: LOW
2~3명: MID
4명: HIGH
5명: ALL
```

#### 영향

- E5, F2와 엔딩에서 5명 완료 장면이 `HIGH` 또는 `ALL`로 다르게 호출된다.
- 이리스의 `public` 고백 판정이 저장·로드 후 달라질 수 있다.
- `settlement_tier` enum과 실제 장면 ID가 일치하지 않는다.

#### 권장 대처안

현재 흐름도와 상태표에서 이미 사용하는 4단계를 채택한다.

```text
LOW | MID | HIGH | ALL
```

- 5명 완료는 `settlement_tier=ALL`.
- `all_servants_complete=true`는 별도 파생 불리언으로 유지한다.
- J4의 `BASE/EXPANDED/FULL`은 연구원 기록 수 판정이므로 관계 결산 enum과 분리한다.

### 3.5 마라 2 병합·분리 결과 저장 누락

**심각도: 중간**

#### 문제

[08 메인 퍼즐](../../08_이벤트상세_03_메인퍼즐.md)과 [12 사용인 핵심 관계](../../12_이벤트상세_07_사용인핵심관계.md)는 E3_5 종료 시 다음 선택을 제공한다.

- 감정 주석 병합.
- 원본·주석 분리 보존.

[14 엔딩](../../14_이벤트상세_09_엔딩.md)은 이 결과에 따라 마라 2의 엔딩 반응을 다르게 출력한다. 그러나 `17`의 E3_5 완료 효과에는 `archive_resolution`을 실제로 쓰는 항목이 없다.

#### 영향

- 즉시 장면에서는 선택이 반영되지만 저장·로드 뒤 `none`으로 돌아갈 수 있다.
- 현실·잔류 엔딩에서 병합과 분리 보존 대사가 잘못 출력될 수 있다.

#### 권장 대처안

E3_5 완료 효과에 다음을 추가한다.

```yaml
set_from_choice:
  servants.MARA2.archive_resolution: selected_resolution
```

허용값:

```text
none | merged | separated
```

`E3_5_complete`, `REC_MARA2`, 관계 변화와 같은 트랜잭션에서 저장한다.

### 3.6 검은 거울과 E3_5 위치 ID 불일치

**심각도: 중간**

#### 문제

[05 공간 구성 지도·동선](../../05_공간구성지도_및_동선.md)은 검은 거울을 대응접실 남쪽의 별도 공간인 `M1_MIRROR_GALLERY`에 둔다. 그러나 일부 문서는 아직 다음 표기를 사용한다.

- 대응접실 거울.
- 거울 복도.
- 위치 ID 없는 검은 거울.

또한 `17`의 E3_5는 지도에 없는 `NORTH_ARCHIVE_COLOR_ROOM`을 `location_id`로 사용한다.

#### 영향

- 검은 거울이 대응접실 내부 오브젝트로 잘못 배치될 수 있다.
- C0~C5의 출입 잠금과 에드가 제지 지점이 달라질 수 있다.
- E3_5 장면 로드 시 존재하지 않는 위치 ID를 참조할 수 있다.

#### 권장 대처안

공식 위치를 다음과 같이 통일한다.

```text
C0~C5: M1_MIRROR_GALLERY
E3_5 전반: H0_COLOR_SEPARATION
E3_5 후반: H0_PERSONALITY_ARCHIVE
```

`거울 복도`는 `거울 회랑`으로, `대응접실 거울`은 `대응접실 남쪽 거울 회랑의 검은 거울`로 교정한다.

### 3.7 Godot 데이터 계약 미동기화

**심각도: 중간**

#### 문제

`17`은 고정된 구현 계약을 표방하지만 최신 상세 문서에서 요구하는 다음 상태를 충분히 반영하지 않았다.

- `reset_state`.
- `failure_record`.
- `route_snapshot`.
- `shortcut_resume`.
- `iris_confession_state`.
- 퍼즐별 `active/resolved/superseded` 실패 상태.
- P4 하위 장면과 objective group.

추가 충돌:

- `camouflage_filter_enabled`와 `camouflage_filter_disabled`가 함께 사용된다.
- `final_decision_unset`은 선언되지 않은 가상 플래그다.
- `all_servants_complete`는 로드 시에만 재계산한다고 적혀 있어 플레이 중 E3 완료 직후 캐시가 낡을 수 있다.
- E3_5의 `interaction_nodes`가 빈 배열이다.

#### 영향

- 상세 기획을 그대로 데이터 리소스로 옮길 수 없다.
- 세이브 복구와 숏컷 재진입이 구현자 판단에 따라 달라진다.
- 반대 극성의 위장 필터 불리언이 동시에 참이거나 거짓인 모순 상태가 생길 수 있다.

#### 권장 대처안

`17`을 다음 순서로 다시 동기화한다.

1. `02`의 시스템 상태와 후반 휴식 규칙.
2. `03`의 진행·분기 순서.
3. `04`의 이벤트·상태 입출력.
4. `05`의 공식 `location_id`.
5. `07`의 리셋 트랜잭션과 숏컷 재개 상태.
6. `08`의 퍼즐·실패 상태.
7. `12~14`의 관계·고백·엔딩 변형.

위장 필터는 단일 enum 사용을 권장한다.

```text
ACTIVE | DISABLED | BROKEN
```

`final_decision_unset`은 별도 플래그로 만들지 않고 `final_decision == unset` 조건식으로 처리한다.

### 3.8 통합본과 구식 용어

**심각도: 낮음**

#### 문제

다음 구식 또는 축약 표기가 남아 있다.

- 단일 `RESET`.
- `거울 복도`.
- P3와 B2를 구분하지 않은 `서재`.
- 5명 완료를 HIGH로 처리하는 결산 표.
- F0-E의 세 비구속적 의향과 `final_choice_relation` 생략.

[18 통합본](../../18_GGB_게임기획서_v0.4_통합본.md)은 참조 무결성이 일치한다고 선언하지만 위 충돌이 남아 있어 현재 상태와 맞지 않는다.

#### 영향

- 통합본만 읽는 작업자가 이전 규칙으로 구현할 수 있다.
- QA 항목과 실제 분기 ID가 달라진다.

#### 권장 대처안

개별 기준 문서를 먼저 수정한 뒤 다음 순서로 요약 문서를 갱신한다.

```text
00 확정사항
→ README
→ 18 통합본
```

단일 `RESET`은 문맥에 따라 다음 중 하나로 치환한다.

- 정상 리셋 체인.
- `NORMAL_RESET`.
- `BROKEN_RESET`.
- `POST_BROKEN_REST`.

## 5. 권장 수정 순서

### 1순위: 진행 불능 제거

1. J4와 E3_4M 순서 통일.
2. D5 이후 `POST_BROKEN_REST` 규칙 확정.
3. 정상 리셋 커밋 순서 통일.

### 2순위: 분기·결산 통일

4. `LOW/MID/HIGH/ALL` 4단계 확정.
5. 마라 2 `archive_resolution` 저장.
6. 이리스 고백 상태와 전원 완료 파생값 재계산 규칙 확정.

### 3순위: 구현 계약 정리

7. 공간 ID와 거울 회랑 명칭 통일.
8. `17`의 상태 스키마와 이벤트 정의 갱신.
9. 리셋·실패·숏컷 복구 QA 추가.

### 4순위: 통합 문서 동기화

10. `00`, `README`, `18`의 용어와 표를 최신화.
11. 상대 링크, 이벤트 ID, 상태 플래그, Mermaid를 다시 검사.

## 6. 완료 판정

다음 조건을 모두 만족하면 본 검토에서 확인한 충돌이 해결된 것으로 본다.

- 관계 이벤트 0명 경로가 `J4 → E3_4M → E5 → E6`으로 진행된다.
- 최초 BROKEN_RESET 이후 수면으로 E구간 수리 상태가 초기화되지 않는다.
- `SYS_COMMIT`과 `SYS_MEMORY` 실패 후 중복 보상 없이 재개된다.
- 관계 완료 5명은 모든 문서에서 `ALL`로 판정된다.
- 마라 2 병합·분리 선택이 저장·로드 후 유지된다.
- 모든 실제 `location_id`가 `05`의 레지스트리에 존재한다.
- `17`이 `04`, `07`, `08`의 필수 상태를 모두 선언한다.
- `18`이 개별 기준 문서와 동일한 용어와 분기값을 사용한다.
