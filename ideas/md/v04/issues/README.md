# GGB v0.4 충돌·오류 관리

## 1. 목적

이 폴더는 v0.4 기획 문서에서 발견한 충돌, 오류, 해결 상태와 검증 이력을 관리한다.

기획 본문과 문제 추적 문서를 분리하여 다음 사항을 명확하게 확인한다.

- 현재 해결해야 할 문제.
- 수정 중인 문제.
- 문서 수정은 끝났지만 검증이 필요한 문제.
- 해결과 회귀 검증이 완료된 문제.
- 보류하거나 의도적으로 수정하지 않는 문제.

## 2. 현재 문서

| 문서 | 역할 | 사용 기준 |
| --- | --- | --- |
| [충돌·오류 레지스트리](충돌오류_레지스트리.md) | 전체 상태 집계와 개별 이슈 색인 | 현재 운영 대시보드 |
| [개별 이슈](items/README.md) | ID별 상태, 영향, 해결안, 체크리스트, 검증 이력 | 각 이슈의 기준 문서 |
| [초기 검토 보고서](archive/초기_검토보고서.md) | ID 체계 도입 전 최초 검토 결과 | 변경 이력 참고 |

## 3. 폴더 구조

```text
issues/
├─ README.md
├─ 충돌오류_레지스트리.md
├─ items/
│  ├─ README.md
│  ├─ GGB-CNF-2026-0001_제목.md
│  ├─ ...
│  └─ GGB-ERR-2026-0007_제목.md
└─ archive/
   └─ 초기_검토보고서.md
```

향후 관련 문서는 다음 기준으로 배치한다.

| 종류 | 위치 |
| --- | --- |
| 전체 이슈 집계와 탐색 | `충돌오류_레지스트리.md` |
| 현재 이슈 상태와 해결 계획 | `items/`의 ID별 파일 |
| 자동·수동 검사 결과 | `validation/` |
| 결정 기록 | `decisions/` |
| 해결 전 원본 보고서 | `archive/` |
| 상태 집계용 데이터가 필요할 경우 | `data/` |

`validation`, `decisions`, `data` 폴더는 실제 문서가 생길 때 생성한다.

## 4. 상태 확인

현재 상태는 [충돌·오류 레지스트리의 상태 대시보드](충돌오류_레지스트리.md#3-상태-대시보드)를 기준으로 한다.

상태 의미:

```text
OPEN
→ IN_PROGRESS
→ RESOLVED
→ VERIFIED
```

- `OPEN`: 해결 필요.
- `IN_PROGRESS`: 수정 중.
- `RESOLVED`: 문서 수정 완료, 검증 전.
- `VERIFIED`: 검증 완료.
- `BLOCKED`, `DEFERRED`, `WONT_FIX`: 별도 사유 기록 필요.

## 5. 관리 규칙

1. 충돌은 `GGB-CNF-YYYY-NNNN` ID를 사용한다.
2. 오류는 `GGB-ERR-YYYY-NNNN` ID를 사용한다.
3. 발급한 ID는 삭제하거나 다른 문제에 재사용하지 않는다.
4. 파일명은 `ID_제목.md` 형식을 사용한다.
5. 개별 이슈 파일을 상태와 해결 내용의 기준으로 삼는다.
6. 해결된 항목도 레지스트리와 `items/`에서 제거하지 않는다.
7. 상태를 `RESOLVED`로 변경할 때 수정 문서와 해결 내용을 기록한다.
8. 상태를 `VERIFIED`로 변경할 때 검증 ID와 날짜를 기록한다.
9. 개별 파일 상태를 바꾸면 레지스트리 대시보드도 같은 상태로 갱신한다.
10. 기준 문서를 먼저 수정하고 `00`, v04 `README`, `18`은 마지막에 동기화한다.
11. 과거 보고서는 `archive/`에서 읽기 전용 기록으로 보존한다.
