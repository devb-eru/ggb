# GGB 프로젝트 역할

## 1. 역할 배정

각 역할에서 첫 번째로 기재된 사람이 팀장이다. 팀장은 영역 내 작업 배정, 우선순위 제안, 검토 조율을 맡으며 프로젝트 관리자 `beru`가 영역 간 충돌과 최종 운영 결정을 조정한다.

| 영역 | 팀장 | 구성원 | 시간 배분 | 주요 책임 |
| --- | --- | --- | ---: | --- |
| 프로젝트 관리 | `beru` | - | 별도 산정 | 우선순위, 마일스톤, 담당 배정, 차단 해소, 운영 결정 |
| 기획 | `gatam` | `niik`, `beru` | 개인 공통 가용시간에서 배분 | 기획 정본, 범위, 문서 간 정합성 |
| 아트 | `210` | `nana` | 개인 공통 가용시간에서 배분 | 게임 아트, UI 시각 자산, Steam 상점 이미지 |
| 사운드 | `NOne` | - | 개인 공통 가용시간에서 배분 | 음악, 효과음, 오디오 품질 |
| 개발 | `beru` | `niik` | 개인 공통 가용시간에서 배분 | Godot 구조, 구현 데이터, 빌드 |
| QA | `gatam` | `beru` | 개인 공통 가용시간에서 배분 | 완료 조건, 회귀 검사, 공개 빌드 승인 |
| Steamworks·릴리즈 | `beru` | - | 개발·운영 가용시간 내 | 상점, 데모 제출, 행사 등록, 배포 |

모든 담당자는 주 7일, 하루 8시간 이상 작업 가능하므로 개인별 주간 가용 상한은 최소 56시간이다. 고정 휴무일은 없고 배정된 일이 없을 때 쉰다. 한 사람이 여러 역할을 맡아도 시간은 역할별로 누적하지 않고 하나의 개인 공통 풀에서 배분한다.

56시간 전부를 신규 제작에 미리 채우지 않는다. 기본 계획 부하는 80% 이내인 주 44시간 48분으로 두고, 나머지 20%는 회의, 검토, 수정 대응, 긴급 작업과 휴식에 사용한다. 마일스톤 긴급 상황에서는 실제 가용 범위 안에서 재배분한다.

## 2. 의사결정과 검토

- 프로젝트 관리자: `beru`
- 영역 내 1차 판단: 해당 팀장
- 여러 영역에 영향을 주는 결정: 해당 팀장들과 협의 후 `beru` 승인
- 별도 검토자가 있으면 작업 담당자와 검토자를 분리한다.
- 인력상 분리가 어렵다면 자기 검토를 허용하되 체크리스트와 QA 근거를 남긴다.
- Steam 제출 빌드와 외부 공개 빌드는 가능하면 작업 담당자 외 한 명이 재검증한다.

개인 이메일과 Discord 계정은 공개 저장소에 직접 기록하지 않고 비공개 운영 설정이나 외부 협업 도구에서 관리한다.

### 3.1 GitHub 대리 반영

`gatam`, `210`, `nana`, `NOne`이 Discord로 전달한 결과물은 `beru`(`devb-eru`)가 저장소에 반영한다.

- 작업 항목이나 PR 설명에 실제 제작자, Discord 전달 메시지와 전달일을 기록한다.
- 커밋 작성자가 `beru`여도 제작자를 `contributor: 이름`으로 남긴다.
- `beru`는 파일 출처, 사용 권리, 대상 작업 ID와 기본 동작 여부를 확인한 뒤 반영한다.
- 수정 요청과 승인 결과는 원 작업 항목에 연결한다.
- Discord만 원본 보관소로 사용하지 않는다. 수신한 원본은 Git/LFS 또는 별도 백업 위치에 보존한다.

## 4. 현재 기획 이슈 기본 배정

- 구현 데이터, 저장, 상태 스키마: 개발 팀장 `beru`
- 서사·관계 판정, 문서 용어 정합성: 기획 팀장 `gatam`
- 참여자가 필요하면 팀장이 구성원에게 세부 작업을 나누되, 이슈의 최종 책임자는 한 명만 유지한다.
