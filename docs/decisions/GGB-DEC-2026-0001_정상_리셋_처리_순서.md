## 결정해야 할 질문

정상 리셋에서 영구 상태 커밋과 사용인 기억 압축을 어떤 순서와 실패 경계로 처리할 것인가.

## 배경과 제약

- `07`은 `SYS_COMMIT → SYS_MEMORY` 순서를 공식 체인으로 사용한다.
- `02`는 사용인 기억 압축 뒤 영구 상태를 커밋하는 반대 순서를 사용한다.
- 두 단계 중간에 종료되어도 중복 적용이나 물리 상태의 조기 폐기가 발생하면 안 된다.
- [GGB-ERR-2026-0002](../../ideas/md/v04/issues/items/GGB-ERR-2026-0002_리셋_트랜잭션_복구_상태_스키마_누락.md)의 복구 스키마는 이 결정에 의존한다.

## 선택지

| 선택지 | 내용 | 장점 | 단점·위험 |
| --- | --- | --- | --- |
| A | `SYS_COMMIT → SYS_MEMORY`를 정본으로 채택하고 두 단계가 성공한 뒤 물리 상태를 폐기 | 현재 `07`과 레지스트리 권장안에 일치, 단계별 재개 지점이 명확 | commit 후 memory 전 종료를 복구할 transaction 상태 필요 |
| B | `SYS_MEMORY → SYS_COMMIT`을 정본으로 채택 | `02`의 기존 설명 유지 | `07`과 관련 흐름을 다시 수정해야 하며 기억 적용 후 commit 실패 복구가 필요 |
| C | 두 처리를 하나의 원자적 복합 단계로 추상화 | 외부 흐름이 단순해짐 | Godot 저장에서 실제 원자성 보장이 어렵고 내부 재개 상태는 여전히 필요 |

## 결정

**A를 채택한다.** 정상 리셋은 `SYS_COMMIT → SYS_MEMORY` 순서로 처리하고, 두 단계가 성공한 뒤에만 현재 루프의 물리 상태를 폐기한다.

## 선택 이유

`07`을 공식 체인으로 이미 사용하고 있으며, 단계별 transaction ID와 멱등 처리를 추가하면 중간 실패도 명시적으로 복구할 수 있다.

## 결과와 영향

- A 채택 시 `02`의 순서를 교정한다.
- `17`에 transaction ID, 현재 phase, 단계별 완료 여부와 재개 규칙을 추가한다.
- `QA-CNF-0003-ORDER`, `QA-CNF-0003-CRASH`, `QA-ERR-0002-PHASE`, `QA-ERR-0002-IDEMPOTENT`를 검증한다.
